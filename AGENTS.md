# AGENTS.md

This repository is a small Python CLI app for asset allocation configuration
and anti‑FOMO portfolio monitoring. The guidance below is for agentic coding
tools working in this repo.

## Quick Start Commands

- Install dependencies:
  - `pip install -r requirements.txt`
- Single run (reads `config.yaml` and optional `config.asset.yaml`):
  - `python run.py`
- Asset configuration wizard (generates `config.asset.yaml`):
  - `python run.py configure`
- Continuous scheduler (daily run at configured time):
  - `python run.py schedule`
- Demo mode (mock data, no network):
  - `python demo.py`

## Build/Lint/Test

There is no formal build system, linter, or unit test suite.

- Build: N/A

## Runtime Notes

- The CLI entrypoint is `run.py`.
- The main app class is `AntiFOMO` in `src/main.py`.
- Asset configuration is generated by `src/asset_configurator.py`.
- Market data is fetched via `akshare` (network required).
- AI analysis is optional; relies on OpenAI‑compatible API.

## Configuration Files

- `config.yaml`: main config; should avoid secrets.
- `config.asset.yaml`: generated by wizard; used to override holdings.
- `config.example.yaml`: example config for users.

Environment variables (preferred for secrets):
- `OPENAI_API_KEY` or `AI_API_KEY`
- `OPENAI_BASE_URL` (for compatible providers)

## Code Style Guidelines

### General

- Language: Python 3.x.
- Use ASCII in source files unless file already uses Unicode.
- Favor small, readable functions over deep abstraction.
- Avoid unnecessary comments; explain only non‑obvious behavior.

### Imports

- Standard library first, then third‑party, then local imports.
- Keep imports at top of file.
- Use explicit imports (no wildcard).

### Formatting

- Indentation: 4 spaces.
- Use double quotes for strings in code (matches current codebase).
- Keep line length reasonable (≈ 100–120 chars).
- Prefer f‑strings for formatting.

### Types

- Use type hints for public functions and dataclasses.
- Prefer `Optional[T]` for nullable values.
- Dataclasses are used for data containers (`@dataclass`).

### Naming Conventions

- Classes: `CamelCase`.
- Functions/variables: `snake_case`.
- Constants: `UPPER_SNAKE_CASE`.
- File names: `snake_case.py`.

### Error Handling

- Prefer graceful failure with clear console messages.
- For external calls (network/API), wrap in try/except and return safe defaults
  (e.g., empty DataFrame, `None`) where appropriate.
- Avoid raising exceptions for user‑supplied config unless the system cannot
  proceed. Use error messages instead.

### IO and Side Effects

- Keep configuration parsing in `src/main.py` and `src/asset_configurator.py`.
- Network access is centralized in `src/data_fetcher.py`.
- Notification output is centralized in `src/notification.py`.
- Avoid writing files outside explicit config outputs.

### AI Analysis

- `src/ai_analyzer.py` is the only module that should call LLM APIs.
- Do not hardcode API keys; use environment variables or config.
- If adding providers, keep a consistent interface and config keys.

### Asset Configuration Wizard

- Keep the interactive workflow simple and resilient to invalid input.
- New categories should be easy to add in `_default_categories()`.
- Generated config must remain YAML‑serializable and human‑readable.

## Repository Conventions
