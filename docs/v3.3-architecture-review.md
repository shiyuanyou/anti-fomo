# v3.3 架构审查报告与技术债务分析

## 执行摘要

当前v3.3实现存在多处架构问题，需要在后续开发前进行重构，以避免技术债积累。

**风险等级**: 🔴 高
**建议**: 在继续v3.4功能开发前，先进行架构重构

---

## 1. 发现的主要问题

### 1.1 数据模型不一致 🔴 **严重**

**问题描述**:
目前有三种不同的模板数据结构：

1. **`web/public/templates.json`** (前端期望格式):
```json
{
  "id": "global_balanced",
  "name": "全球均衡型",
  "description": "...",
  "metrics": { "expected_return": 14.91, ... },
  "allocation": { "A股大盘": 20.0, ... },
  "personality_tags": [...],
  "original_data": { ... }  // 完整的原始数据
}
```

2. **`api/models.py`** (数据库模型):
```python
class Template(Base):
    id = Column(String(20))           # AF-T001 (新格式)
    name = Column(String(100))
    description = Column(Text)
    allocation = Column(JSON)         # {symbol: weight}
    metrics = Column(JSON)
    tags = Column(JSON)
```

3. **`src/template_engine/templates.py`** (旧Python格式):
```python
@dataclass
class PortfolioTemplate:
    id: str                           # global_balanced
    name: str
    tagline: str                      # ❌ 数据库缺少
    target_audience: str              # ❌ 数据库缺少
    risk_level: str                   # ❌ 数据库缺少
    allocations: List[AssetAllocation] # 复杂结构
    metrics: TemplateMetrics
    personality_tags: List[str]
    personality_description: str
```

**影响**:
- 前端调用 `/api/templates` 期望得到 `global_balanced` 格式的数据
- 但后端返回的是 `AF-T001` 格式，缺少 `tagline`, `risk_level`, `target_audience` 等字段
- 导致前端无法正确显示模板详情

**数据字段对比**:
| 字段 | JSON | DB Model | Python Class |
|------|------|----------|--------------|
| id | global_balanced | AF-T001 | global_balanced |
| tagline | ✅ | ❌ | ✅ |
| risk_level | ❌ | ❌ | ✅ |
| target_audience | ❌ | ❌ | ✅ |
| original_data | ✅ | ❌ | ❌ |
| allocations (复杂) | ❌ | ❌ | ✅ |
| allocation (简单) | ✅ | ✅ | ❌ |

### 1.2 API目录结构不合理 🟡 **中等**

**当前结构**:
```
api/
├── main.py      # 111行，包含所有路由
├── models.py    # 67行，所有模型
├── schemas.py   # 59行，所有schema
└── crud.py      # 79行，所有CRUD
```

**问题**:
- 平级结构，没有模块化
- `main.py` 随着功能增加会越来越臃肿
- 缺少分层：routers/, services/, config/
- 所有代码混在一起，难以维护

**对比标准FastAPI项目结构**:
```
api/
├── app/
│   ├── main.py          # 仅应用配置
│   ├── config.py        # 环境配置
│   └── dependencies.py  # 依赖注入
├── routers/
│   ├── templates.py     # 模板路由 (~50行)
│   ├── shares.py        # 分享路由 (~50行)
│   └── assets.py        # 资产配置路由 (~30行)
├── services/
│   ├── template_service.py
│   └── share_service.py
├── models/
│   ├── template.py
│   └── share.py
└── schemas/
    ├── template.py
    └── share.py
```

### 1.3 代码重复与依赖混乱 🟡 **中等**

**问题**:
1. `src/template_engine/templates.py` 定义了完整的模板数据结构
2. `api/models.py` 重新定义了简化的结构
3. `scripts/migrate_templates.py` 需要复杂的转换逻辑来桥接两者
4. `web/public/templates.json` 又是一种导出格式

**后果**:
- 修改模板结构需要同时改3-4个地方
- 容易遗漏，导致数据不一致
- 新人难以理清数据源在哪里

### 1.4 前端数据获取方式不统一 🟡 **中等**

**问题**:
- `TemplatesView.vue`: `fetch('/api/templates')`
- `CompareView.vue`: `fetch('/api/templates')`
- 但目前返回的数据格式与前端期望不符
- 前端还在使用 `templates.json` 静态文件作为fallback

**影响**:
- 无法真正测试前端与后端的集成
- 需要同时维护静态JSON和API两种数据源

### 1.5 数据库模型设计缺陷 🟠 **轻微**

**问题**:
- `allocation` 存储为简单 JSON `{symbol: weight}`
- 但前端需要更详细的分配信息（类别、区域等）
- `metrics` 没有定义完整的字段结构
- 缺少索引优化

## 2. 重构建议

### 2.1 统一数据模型 🔴 **必须**

**方案A: 采用 templates.json 格式作为标准** (推荐)

理由：
- 前端已经基于此格式开发
- 包含完整的模板信息（tagline, risk_level等）
- 与旧Python数据结构兼容

**数据库模型更新**:
```python
class Template(Base):
    __tablename__ = "templates"
    
    id = Column(String(30), primary_key=True)      # global_balanced
    name = Column(String(100), nullable=False)
    tagline = Column(String(200))                   # 新增
    description = Column(Text)
    target_audience = Column(String(200))           # 新增
    risk_level = Column(String(10))                 # 新增
    
    # allocation 存储为列表，支持复杂结构
    allocations = Column(JSON)  # [{"category": "A股大盘", "region": "中国", "weight": 0.2}]
    
    metrics = Column(JSON)      # {expected_return, volatility, max_drawdown, sharpe_ratio, data_period}
    personality_tags = Column(JSON)  # 重命名
    
    # 可选：存储完整原始数据以兼容
    original_data = Column(JSON)
```

**迁移策略**:
1. 删除 `scripts/migrate_templates.py`
2. 新建 `scripts/init_db_from_json.py` 直接从 `templates.json` 导入
3. 保持 `templates.json` 作为唯一数据源

### 2.2 重构API目录结构 🟡 **推荐**

**新结构**:
```
api/
├── app/
│   ├── __init__.py
│   ├── main.py              # 精简到 ~30行
│   ├── config.py            # 配置管理
│   └── dependencies.py      # get_db() 等依赖
├── routers/
│   ├── __init__.py
│   ├── templates.py         # 模板路由
│   ├── shares.py            # 分享路由
│   └── assets.py            # 资产配置路由
├── services/
│   ├── __init__.py
│   ├── template_service.py  # 业务逻辑层
│   └── share_service.py     # 业务逻辑层
├── models/
│   ├── __init__.py
│   ├── database.py          # 数据库连接
│   ├── template.py          # 模板模型
│   └── share.py             # 分享模型
├── schemas/
│   ├── __init__.py
│   ├── template.py          # 模板schema
│   └── share.py             # 分享schema
└── crud/
    ├── __init__.py
│   ├── template.py          # 模板CRUD
│   └── share.py             # 分享CRUD
```

**实施步骤**:
1. 创建新的目录结构
2. 逐步迁移代码，保持功能不变
3. 每迁移一个模块后测试
4. 删除旧的平级文件

### 2.3 废弃旧代码 🟡 **必要**

**需要废弃的文件**:
1. `serve.py` → 重命名为 `serve_legacy.py` 或删除
2. `src/template_engine/templates.py` → 标记为废弃
3. `web/public/templates.json` → 作为备份保留
4. `scripts/migrate_templates.py` → 删除

**过渡策略**:
1. 在文档中明确标注哪些已废弃
2. 在代码中添加 `DeprecationWarning`
3. 保留1-2个版本后完全删除

### 2.4 统一错误处理和日志 🟠 **推荐**

**当前问题**:
- 错误处理不一致
- 没有统一的日志配置
- 缺少API文档描述

**改进方案**:
```python
# api/app/exceptions.py
class TemplateNotFoundError(Exception):
    pass

class ShareNotFoundError(Exception):
    pass

# 全局异常处理
@app.exception_handler(TemplateNotFoundError)
async def template_not_found_handler(request, exc):
    return JSONResponse(
        status_code=404,
        content={"error": "Template not found", "detail": str(exc)}
    )
```

## 4. 不重构的风险

如果继续在当前架构上开发v3.4：

1. **数据不一致加剧**: 每次修改模板结构都需要改4+个文件
2. **前端Bug增多**: 后端API返回格式与前端期望不匹配
3. **新人上手困难**: 代码结构混乱，难以找到正确位置
4. **维护成本指数增长**: 平级结构无法支撑更多功能
5. **测试困难**: 没有清晰的模块边界，难以单元测试

---

## 5. 结论与建议

**强烈建议立即进行架构重构**，原因：

1. **技术债务可控**: 现在代码量还不大（~400行），重构成本低
2. **功能依赖**: v3.4的分享功能需要稳定的API基础
3. **团队效率**: 清晰的架构能提高后续开发效率
4. **质量保障**: 分层架构更容易测试和维护

**下一步行动**:
1. 确认是否进行重构
2. 如果确认，按Phase 1-4顺序执行
3. 每个Phase完成后进行Code Review
4. 全部完成后更新AGENTS.md文档
